# Review Code

Используй этот skill, когда нужно сделать инженерное ревью изменений, а не просто кратко пересказать diff.

## Когда применять

- пользователь просит сделать review или code review;
- нужно проверить готовый diff, коммит, PR или локальные изменения;
- перед закрытием задачи нужно оценить риски и незавершенности;
- требуется независимая проверка изменений перед merge или release.

## Workflow

1. Определи контекст ревью:
   какие файлы изменены, какой был ожидаемый результат, какие зоны риска затронуты.
2. Ищи в первую очередь:
   баги, регрессии, несоответствие контрактам, пропущенные edge cases, сломанные инварианты.
3. Затем проверь:
   архитектурные риски, избыточную сложность, небезопасные допущения, расхождения документации и кода.
4. Отдельно проверь:
   хватает ли тестов, покрыты ли error states, миграционные сценарии, rollback и совместимость.
5. Если находок нет, явно скажи, что критичных findings не найдено, и перечисли остаточные риски или пробелы проверки.

## Output

Возвращай:

- findings в порядке убывания серьезности;
- для каждого finding: что не так, почему это риск, где это проявится;
- список открытых вопросов или допущений;
- только после этого краткий summary.

## Guardrails

- Не сводись к пересказу измененных файлов.
- Не ставь стиль и вкусовые замечания выше функциональных рисков.
- Если данных недостаточно для уверенного вывода, явно пометь это как ограничение ревью.
- Если пользователь просит именно review, findings должны быть основной частью ответа.
