# Docs Closeout

Закрывай задачу через проверку документации, а не только через перечисление измененных файлов.

## Когда применять

- задача завершена или почти завершена;
- были изменения в коде, структуре проекта, архитектуре, интерфейсе, данных, инфраструктуре или правилах работы;
- нужно понять, достаточно ли обновлена документация перед финальным ответом или коммитом.

## Workflow

1. Определи, что именно изменилось:
   код, структура, архитектура, интерфейс, данные, процессы, инфраструктура, документация.
2. Проверь, достаточно ли обновить только changelog.
3. Отдельно проверь, нужно ли обновить:
   - корневой `README.md`;
   - ближайший к изменению `README.md`;
   - task docs или рабочие заметки проекта;
   - архитектурную документацию, ADR, схемы, `docs/architecture.*`;
   - API, deployment, operations или onboarding docs;
   - `AGENTS.md`, если менялись правила работы агентов или проектные инструкции.
4. Если документ не обновлялся, но должен был, отметь это как незавершенность.
5. Подготовь короткий итог работы и предложи один commit message на русском.

## Output

Возвращай:

- короткий итог результата;
- список документов, которые были обновлены;
- список документов, которые еще нужно обновить;
- явное указание, делался ли commit;
- один предложенный commit message на русском.

## Guardrails

- Не требуй лишних обновлений для trivial-изменений, если реальная документация не затронута.
- Не выполняй `push`, `gh`, CI-команды и релизные действия без прямой команды пользователя.
- Не создавай коммит автоматически.
- Если изменение затрагивает уже зафиксированные правила работы, напомни про обновление `AGENTS.md` и соседней релевантной документации.
